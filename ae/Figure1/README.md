## Figure 1(a): FIO on HDD

### Testbed

1 CloudLab c220g5 machine.

### Steps

> **Note:** Run `bash plot_env.sh` (in the repo root) and `source ~/xk-py/bin/activate` before plotting.

```bash
bash install_1a.sh
export KERNEL_DIR=~/linux-6.8.0     # kernel source for codegen
bash run1a.sh                       # ~3 minutes
python3 plot/plot_1a.py             # → plot/figure1a.pdf
```

**Machine time:** ~3 minutes &nbsp;|&nbsp; **Human time:** ~1 minute

### Expected Results

> **Note on workload:** The I/O log (`iolog/`) is regenerated by `run1a.sh`
> using a fixed random seed. The access pattern follows the same regionalized
> random 4K layout as the original paper, but the exact iolog used in the paper
> is no longer available. Results may differ slightly in absolute numbers;
> however the performance improvement trend is consistent across runs.

Xkernel tunes `BLK_MAX_REQUEST_COUNT` from its default value 32 to 128 at
runtime. With a larger request count the block layer can merge more I/O
requests before dispatching, dramatically improving throughput on the HDD.

Representative results on CloudLab c220g5 (kernel 6.8.0-101-generic):

| Workload | V=32 | V=128 | Improvement |
|----------|------|-------|-------------|
| Read     | 27.15 MB/s | 110.04 MB/s | **+305%** |
| Write    | 6.12 MB/s  | 97.68 MB/s  | **+1496%** |

The exact numbers may vary by ±10–15% between runs depending on disk state and
system load, but the trend (>3× read improvement, >10× write improvement) should
be consistent.
